F · Reference Implementation Blueprint

From principles to a production codebase

A repo where domain logic is a pure Python package, pipelines are declarative Lakeflow assets, configuration is metadata not code, and a new domain ships as a bundle. Plus the medallion layout, catalog strategy, multi-domain plan, and the functional + non-functional design.

The structural invariant

mdm_core has zero dependency on PySpark, Delta, or the Databricks SDK — enforced by a CI import-linter rule. That single constraint is what operationalizes Dependency Inversion: domain rules are unit-tested in milliseconds on pandas and run unchanged on a billion rows because the Spark-ness lives only in mdm_runtime.

Catalog, schema & medallion design

Environment separation by catalog; domain + tier separation by schema. This makes UC grants, lineage, and promotion clean.

catalog            schema                  example tables
────────           ──────                  ──────────────────────────────────
mdm_dev    │
mdm_test   │  ┌─ <domain>_bronze        raw_records, _quarantine
mdm_prod   ┼──┤  <domain>_silver        standardized_records, dq_results
           │  │  <domain>_master        clusters, golden_records, survivor_attr,
           │  │                          xref, merge_lineage         ← Master/Trust
           │  ├─ <domain>_gold          mart_*, hierarchy_*, serving_*
           │  ├─ mdm_steward            cases, decisions, assignments  (⇄ ext UI)
           │  ├─ mdm_reference          code_sets, mappings, crosswalks
           │  └─ mdm_ops                run_audit, monitor_metrics, config_*
<domain> ∈ { customer, product, supplier, location }

Naming: snake_case · table = noun_plural · views = vw_* · config = cfg_*
        every table tagged: domain=, tier=, pii=, owner=  (UC tags)
Why catalog-per-env beats schema-per-env

Catalog boundaries map cleanly to UC isolation, Delta Sharing scopes, and bundle targets. dev→test→prod promotion is a bundle target switch; cross-env reads are blocked by catalog ACLs, not by hopeful naming. Domain-per-schema then keeps per-domain stewardship and grants independent — true tenant/domain isolation.

Multi-domain support

Customer, product, supplier, location, and reference data share one engine, many configs. A domain is a YAML bundle (canonical schema + standardization + blocking + match strategies + survivorship + hierarchy), never a new codebase. Domain-specific quirks (product taxonomies, supplier DUNS, address geocoding) are plugged in as strategies implementing the same ports.

Batch + near-real-time, one model

 BATCH (nightly / on-demand)                NEAR-REAL-TIME (continuous)
 ─────────────────────────────             ───────────────────────────────
 Auto Loader file trigger        ───┐  ┌── Structured Streaming / Lakeflow Connect CDC
 full/blocked match (Spark)         │  │   incremental match on new+affected blocks
 MERGE mastering                    ▼  ▼   MERGE upserts golden records
 publish snapshot                ┌──────────┐  emit GoldenRecordChanged via CDF
                                 │  SAME     │  → online store / serving for ops lookups
                                 │  mdm_core │
                                 │  domain   │  Same ports, same rules. Only the
                                 │  logic    │  RecordRepository + trigger differ.
                                 └──────────┘

Because domain logic is pure and injected, the same MatchStrategy and SurvivorshipRule run in both modes. Near-real-time only re-resolves the affected blocks (incremental), with a periodic batch reconciliation to catch cross-block drift.

Non-functional architecture (deliverable 8)

Scalability

Blocking keeps match O(n) not O(n²); Photon + autoscaling for billion-row passes; incremental re-match in streaming. Domains scale independently (schema isolation).

Cost efficiency

DBUs + storage only — no per-seat/per-record MDM license. Serverless for spiky steward/serving loads; job clusters for batch; liquid clustering + Z-order on block keys to cut scan.

Security & access control

UC grants on catalog/schema/table; no app-level ACL. Domain isolation by schema; steward roles map to UC groups.

PII protection

UC column masks + row filters; pii= tags drive policy; tokenize at Silver where needed; LLM calls use governed FM endpoints (no data egress).

Observability

system.access/lineage/billing + DLT event log + Lakehouse Monitoring on match rate, merge rate, DQ drift, cost per run.

Resiliency & replayability

Append-only Bronze + idempotent MERGE = safe re-runs. Time travel for point-in-time rebuild. Quarantine instead of pipeline failure on bad data.

Testability

Pure domain unit tests on pandas (ms); contract tests per seam; ephemeral-catalog integration tests in CI. Import-linter forbids Spark in mdm_core.

Tenant / domain isolation

Catalog-per-env, schema-per-domain; per-domain bundles, grants, and steward queues. One noisy domain can't starve another.

CI/CD

GitHub Actions: validate bundle → unit/contract tests → deploy to dev/test → gated prod. Config changes flow through review but don't require code deploy.

Performance tuning

Liquid clustering on block keys; broadcast small reference tables; pandas/Arrow UDFs for vectorized scoring; cache embeddings; batch LLM adjudication of only the gray-zone pairs.
