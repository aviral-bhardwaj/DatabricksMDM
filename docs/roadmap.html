H · Phased Roadmap

MVP → v1 → enterprise scale

Each phase is gated by an architectural milestone, not a calendar. Don't add domains until the engine is config-driven; don't add streaming until batch is idempotent; don't add the LLM adjudicator until you can measure match precision. Build the spine first, then widen.

Phase 0 — Foundations (weeks 0–3)

  • Catalogs (mdm_dev/test/prod), schema-per-domain/tier, UC groups + tags, Asset Bundle skeleton.
  • mdm_core package with ports + dataclasses; import-linter forbidding Spark in core; CI green on pandas unit tests.
  • One source through Bronze via Auto Loader; provenance columns; .nojekyll-style hygiene for repo.
Gate

Domain logic is unit-tested without a cluster; bundle deploys to dev. The DIP boundary is real before any business logic is written.

A source record flows to a governed golden record with explainable survivorship, a steward can override it, the override is audited via CDF, and a BI tool reads the master in-place. That is a complete native loop — narrow but real.

Phase 2 — v1: AI-native matching, multi-domain, near-real-time

  • Semantic match: embed attributes → Vector Search candidate generation; LLM adjudicator (served FM) on the calibrated gray zone only; rationale stored as explanation.
  • Multi-domain: add product, supplier, location as config bundles on the same engine — proving Open/Closed in practice.
  • Near-real-time: Structured Streaming / Lakeflow Connect CDC; incremental re-match of affected blocks; periodic batch reconciliation.
  • Hierarchy: GraphFrames/edge tables; parent/child rollups; soft links from vector similarity.
  • Reference data: mdm_reference with versioned CodeSets + crosswalks feeding standardization.
  • Observability: Lakehouse Monitoring on match precision/recall, merge rate, DQ drift, cost/run; system-table audit dashboards.
  • Online serving: sub-10ms single-record lookups via online store/serving fed from Master.
Gate

Adding the supplier domain requires zero changes to mdm_core — only YAML + a strategy plugin. If it requires editing the engine, Open/Closed was violated; fix it before scaling.

Phase 3 — Enterprise scale & hardening

  • Workflow depth: SLA-driven approval routing, role hierarchies, bulk steward ops, exception queues.
  • Merge/unmerge robustness: unmerge via cluster-history replay + re-survivorship; full bitemporal reprocessing; survivorship A/B against history.
  • Tenant/domain isolation at scale: per-domain bundles, grants, clusters, cost attribution; serverless for spiky loads.
  • Activation breadth: reverse-ETL, Kafka, JDBC adapters; Delta Sharing to partners; data-product publishing to the marketplace.
  • Governance hardening: attribute-based access, column masking on all PII, retention policies, certified lineage end-to-end.
  • Resiliency: idempotent replays, quarantine SLAs, DR via catalog replication; chaos-test re-runs.
  • Match quality program: continuous active-learning loop; precision/recall SLOs per domain; drift alerts.
Gate

End-to-end UC lineage from source to consumption is certifiable; any golden record's full history + decision trail is reconstructable; a new domain onboards in days via config; run cost is attributable per domain. Now it competes with the incumbents on their turf while keeping the native advantages they can't match.

Closing principle

Resist the urge to start with breadth (many domains, every connector) or with the flashy part (the LLM matcher). Start with the narrow native loop and the DIP boundary. Everything valuable — multi-domain, AI matching, enterprise governance — is cheap to add once the engine is pure, config-driven, and the master is a governed Delta object. Build that, and you have a product no Tier-1–3 vendor can structurally replicate.
