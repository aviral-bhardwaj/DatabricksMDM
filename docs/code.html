From blueprint to running code

The blueprint is no longer just a document — it's implemented. The repository ships a real Databricks-native MDM engine: a pure, Spark-free domain core that unit-tests in a tenth of a second, Spark/Delta runtime adapters, and declarative pipelines. This page maps the code to the architecture and shows how to run it.

0pyspark imports in core (enforced)
16→7records → golden (demo, 56% dedup)
4domains by config alone

Three zones, one dependency rule

The codebase enforces the separation the blueprint argues for: domain logic never knows it's on Spark. The dependency arrow points one way — pipelines → runtime → core — so the identical match/survivorship logic runs in a millisecond test and on a billion rows on a cluster.

mdm/core      PURE DOMAIN — no pyspark anywhere (guarded by tests/test_purity.py)
              model.py · ports.py · standardize/ · quality/ · matching/
              (deterministic · probabilistic · semantic · registry) ·
              cluster/connected_components.py · mastering/(survivorship · assembler) ·
              stewardship/workflow.py · config/ · service.py
mdm/runtime   INFRASTRUCTURE — the ONLY zone importing pyspark/delta/SDK
              repository_delta.py · repository_memory.py · encoders.py ·
              sources/(autoloader · jdbc) · adapters/publication.py ·
              uc.py · vector.py · observability.py
mdm/pipelines ORCHESTRATION — declarative, owns no rules
              bronze_ingest.py · silver_standardize.py (DLT) ·
              match_cluster.py · master_publish.py
conf/domains  customer · product · supplier · location  (a domain is data)

Run it yourself — no Databricks needed

pip install -e ".[dev]"

pytest                 # 33 tests, pure-domain, ~0.1s
python -m mdm demo     # end-to-end on bundled SAP/Salesforce/Oracle/Odoo sample data

The demo standardizes, quality-scores, blocks, matches (deterministic + probabilistic), clusters with union-find, and applies rules-as-data survivorship — entirely on pandas via the in-memory repository, which satisfies the same RecordRepository port as the Delta one. Output:

What got fixed from v1

Correctness & security
  • Non-transitive clustering (v1 used monotonically_increasing_id) → union-find connected components, with a test asserting A↔B, B↔C ⇒ one cluster.
  • SQL injection in the v1 audit/lineage/review queries → all identifiers validated against an allowlist and back-quoted; all values bound as parameters (runtime/uc.py).
  • Hardcoded country blocking and customer-only DQ → fully config-driven, any domain.
  • One welded Random Forest on the match path → a pluggable strategy registry (deterministic / probabilistic / semantic).
  • Spark-coupled domain logic → pure core, guarded by an import-purity test.

The superseded v1 remains under legacy/ for reference and migration — see its README for the full before/after.
