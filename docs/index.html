A · Executive Summary

The thesis

A Databricks-native MDM is not an application that runs on Databricks. It is a data product whose system of record, system of governance, system of processing, and system of intelligence are all the lakehouse itself.

The mastering industry was built around a closed hub-and-spoke model: pull data into a proprietary store, master it inside an engine nobody else can see, and push golden records back out through adapters. That topology made sense in 2010. On a lakehouse it is an anti-pattern — it re-creates the very data silo Databricks exists to eliminate, and it strands the most valuable asset in the enterprise (the trusted golden record) outside the catalog that governs everything else.

This blueprint argues for the inverse. The data never leaves. Sources land as Delta. Standardization, matching, survivorship, and publication are declarative pipelines over Delta. The golden record is a Unity Catalog table with row/column security, lineage, and time travel — queryable by every BI tool, notebook, and model in the estate without an export. Stewardship is the only component that legitimately lives outside, and we say exactly why.

The one-sentence definition

Databricks-native MDM = the golden record is a governed Delta/UC object produced by declarative lakehouse pipelines, where match/merge/survivorship are versioned data + open code, governance is Unity Catalog (not a bespoke security model), and intelligence is MLflow/Model-Serving/Vector-Search operating in-place — with zero proprietary mastering store on the critical path.

Five opinions this blueprint will defend

  1. The "Master/Trust" layer is real and Gold is not enough. Bronze/Silver/Gold describes refinement; it says nothing about identity. MDM needs a fourth tier where survivorship, cluster lineage, and trust scores are first-class. We justify it in section D.
  2. Match logic is a strategy, never a hardcode. A Random Forest welded into a matching notebook (as in many starter repos, including the one this product replaces) violates Open/Closed and makes every new domain a code fork. Matching is a pluggable contract.
  3. Stewardship UI belongs outside Databricks — and that's correct, not a compromise. Databricks is a data-and-compute plane, not an OLTP app server. We push case state into Delta and render the workbench in a real web app. Section G defends the boundary.
  4. Semantic matching is the native differentiator. Vector Search + an embedding model + an LLM adjudicator, all in-platform, is something no closed hub can match without exfiltrating your data. This is the moat.
  5. Governance is not a feature you build — it's Unity Catalog you adopt. Re-implementing ACLs, masking, and lineage inside an MDM app is the single clearest signal that a product is not native.
Anti-pattern · the tell-tale sign

If your "Databricks MDM" could be lifted onto Snowflake or Postgres by swapping the JDBC string, it was never native. Native means the architecture cannot be expressed without Delta semantics, UC governance, and declarative lakehouse pipelines.
