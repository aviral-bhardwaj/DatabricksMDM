The MDM landscape, read through a lakehouse lens

Six incumbents define the category. Two insurgents (LakeFusion, Zingg) prove the lakehouse-native thesis is already commercial. Below: a per-vendor teardown, a comparison matrix, and the product lessons — what to copy, avoid, and redesign.

The four archetypes first

Before the vendors, fix the taxonomy. Every product below sits in one of four tiers, and the tier predicts its weaknesses for a Databricks-first buyer more than the brand does.

Per-vendor teardown

Informatica MDM (+ IDMC / CLAIRE)

  • Architecture: Mature multidomain hub. Classic on-prem (Hub Store on Oracle/SQL Server, ActiveVOS workflow) migrating to the IDMC cloud platform. CLAIRE is the cross-product AI/ML engine for matching, classification, and automation.
  • Strengths: Deepest feature surface in the market — match rules, trust framework, hierarchies, reference data (RDM), B2B/B2C, healthcare/pharma accelerators. Enormous connector library. The "nobody got fired for buying it" option.
  • Governance: Its own security/role model, ActiveVOS BPM, audit inside the hub. Powerful but a parallel governance universe to Unity Catalog.
  • Stewardship UX: Mature but heavyweight; the Data Director / IDMC stewardship UI is feature-rich and dated.
  • Entity resolution: Deterministic + probabilistic match with a tunable trust/decay model; CLAIRE ML-assisted match.
  • Weakness for a Databricks-first shop: Heaviest licensing and infra footprint; data is mastered outside the lakehouse; you inherit a second governance plane and a second copy of your data. Tier 1/2.

Reltio

  • Architecture: Cloud-native, multi-tenant SaaS; microservices; Cassandra/Elasticsearch lineage; real-time bidirectional APIs. The reference modern SaaS MDM.
  • Strengths: Real-time graph-based relationships, strong API-first integration, continuous DQ, and FERN (Flexible Entity Resolution Network) augmenting match rules with LLM intelligence. Forrester 2025 Leader. Note: SAP announced its acquisition of Reltio — expect tighter SAP-data gravity going forward.
  • Governance: Vendor-operated; good but opaque — your master lives in Reltio's cloud, not your catalog.
  • Stewardship UX: Best-in-class modern web stewardship and relationship visualization.
  • Entity resolution: Configurable match + FERN LLM augmentation; survivorship via rules.
  • Weakness for a Databricks-first shop: Pure Tier 2. Consumption pricing on a separate platform; data egress + duplication; you cannot run your own models against the master in-place. Openness is the gap.

Semarchy xDM

  • Architecture: Pragmatic, model-driven hub on a standard RDBMS; deploys on-prem or cloud. "xDM" = unified MDM + data quality + reference + workflow.
  • Strengths: Outstanding developer/steward velocity — declarative model, fast app generation, clean match/merge/survivorship (consolidation rule + override rule), ML-assisted matching. Excellent stewardship UX for the price.
  • Governance: Solid built-in workflow, lineage, and history — but again a self-contained model.
  • Entity resolution: ID-match + fuzzy-match into clusters; golden record lifecycle automated from matcher + survivorship config.
  • Weakness for a Databricks-first shop: RDBMS-centric; integrates with the lakehouse (Tier 3) rather than running in it; scale ceiling on very large fuzzy-match workloads vs Spark.

Profisee

  • Architecture: Cloud-native multidomain platform, tight Microsoft/Azure + Purview/Fabric alignment; SQL-Server-rooted.
  • Strengths: Approachable stewardship UI, fast time-to-value, strong matching/survivorship, good Purview integration. Mid-market sweet spot.
  • Weakness for a Databricks-first shop: Microsoft-gravity, not Databricks-gravity; Tier 2/3. On Databricks it is a connected hub, not a resident.

Ataccama ONE

  • Architecture: Unified platform — data quality + catalog + MDM + observability — with a strong AI/ML automation story (profiling, anomaly, auto-matching).
  • Strengths: DQ-first heritage means excellent profiling/standardization; one fabric for catalog + quality + master. Increasingly cloud-flexible.
  • Weakness for a Databricks-first shop: Still a platform you stand up alongside Databricks (Tier 3). Its catalog/quality overlaps and competes with Unity Catalog rather than deferring to it.

SAP MDG (Master Data Governance)

  • Architecture: MDM embedded in S/4HANA; ABAP/HANA; co-deployed (hub) or central governance.
  • Strengths: Unbeatable inside the SAP estate — native data models, validations, and process governance for SAP material/business-partner/finance master. Deep workflow.
  • Weakness for a Databricks-first shop: SAP-centric to the point of gravity well; weak for non-SAP, analytics-led, multi-cloud mastering. Tier 1. (With SAP acquiring Reltio, expect MDG + Reltio convergence aimed at "AI-ready SAP + non-SAP data.")

The insurgents — proof the thesis is commercial

  • LakeFusion — markets itself explicitly as Databricks-native MDM + PIM + Graph: unifies entities, products, and relationships into a governed, AI-ready foundation without moving data out of the lakehouse. This is Tier 4, productized. It validates that "native" is a real, buyable category — and is the closest competitor to this blueprint.
  • Zingg (OSS) — open-source entity resolution that runs inside your Databricks workspace via Spark; the ZINGG_ID lives in Delta, governed by Unity Catalog, trained by your team via active learning. It is not a full MDM (no stewardship/workflow/publishing), but it is the canonical native matching engine — and a perfect "buy the hard part, build the rest" component. Its own framing nails the thesis: "integrating with the warehouse and running natively inside it are architecturally very different things."
Read of the market

The incumbents win on breadth, packaged domains, stewardship maturity, and OOTB connectors. They lose, for a Databricks-first buyer, on openness, data proximity, cost model, and AI-in-place — exactly the axes a Tier-4 product owns. The whitespace is not "a better hub." It is "no hub at all."

Product lessons

Copy

  • Trust/decay framework (Informatica) — survivorship driven by source trust + recency decay, not naive last-write-wins.
  • Graph-native relationships (Reltio) — model relationships as first-class, not as a join afterthought.
  • Model-driven app generation (Semarchy) — config-over-code so a new domain is metadata, not a fork.
  • Consolidation + override survivorship (Semarchy) — automatic value + explicit steward override, both versioned.
  • LLM-augmented matching (Reltio FERN, Informatica CLAIRE) — but run it in-platform.
  • DQ-first standardization (Ataccama) — profile and score before you match.

Avoid

  • A proprietary mastering store on the critical path — the original sin of Tiers 1–3.
  • A parallel governance/security model that duplicates Unity Catalog.
  • Black-box match scoring with no explainability or lineage.
  • An app-server-bound architecture that can't exploit Spark for billion-row fuzzy match.
  • Monolithic licensing that taxes every domain and record.
  • Catalog/quality features that compete with UC instead of deferring to it (Ataccama overlap).
Redesign for the lakehouse

Survivorship → rules-as-data: store rules in Delta config tables, version them, A/B them, and reprocess history with time travel. Match → pluggable strategy contract over Spark, with vector/LLM as just another strategy. Stewardship state → Delta tables read/written by an external UI, so the workbench is replaceable and the audit trail is the data. Publishing → adapters + Delta Sharing / CDF, so "activation" is often zero-copy. Governance → adopt UC wholesale; build nothing.
